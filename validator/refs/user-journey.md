# 用户旅程验证

> **核心原则：先当用户，再当测试员。** 验收不是"这个按钮能不能点"，而是"用户来到这个产品，能不能高效完成他的目标"。

## 1. 产品级用户旅程定义（Step 1 中完成）

在读完 Issue 正文和验收标准后，**在打开浏览器之前**，必须完成以下定义：

### 1.1 产品定位理解

回答三个问题：
1. **这个产品是给谁用的？**（角色：开发者/产品经理/数据分析师/终端用户）
2. **用户来这个产品要完成什么任务？**（核心目标：查看数据/执行操作/阅读文档）
3. **用户的典型使用场景是什么？**（场景：每天看板/偶尔查报告/紧急排查问题）

### 1.2 页面编排评估标准

基于产品定位，评估整体页面结构：

| 维度 | 评估问题 | FAIL 条件 |
|------|---------|----------|
| 导航架构 | 页面数量和分组是否合理？用户能否快速找到目标页面？ | 功能分散在不合理的页面中、关键功能入口隐藏 >2 层 |
| 页面优先级 | 首页/默认页是否展示用户最关心的信息？ | 首页展示次要信息，核心功能需要额外导航才能到达 |
| 操作流程 | 完成核心任务需要几步？是否有不必要的步骤？ | 简单任务需要 >5 步、存在重复操作、需要在页面间来回跳转 |
| 信息流向 | 从概览到详情的信息层次是否清晰？ | 概览和详情混在一起、无法从概览钻取到详情 |
| 页面间一致性 | 各页面的布局模式、交互模式是否统一？ | 页面 A 用 tab 切换，页面 B 用下拉框切换同类操作 |

### 1.3 定义具体用户旅程

为每个核心任务定义 1 条旅程（最多 3 条）：

```
旅程 {N}：{旅程名称}
角色：{谁在用}
目标：{要完成什么}
入口：{从哪个 URL/页面开始}
预期路径：
  1. 打开首页 → 预期看到 {什么信息}，自然会点 {什么}
  2. 进入 {页面} → 预期看到 {什么信息}，预期操作 {什么}
  3. ...
  N. 完成目标 → 预期看到 {确认/结果}
关注点：{这条旅程要验证的核心体验}
```

## 2. 首印象测试（Step 3 Phase A）

导航到每个页面后，**5 秒内**（不滚动、不点击）回答：

| # | 问题 | FAIL 条件 |
|---|------|----------|
| FI-1 | 我能理解这个页面是做什么的吗？ | 页面没有标题、标题不明确、内容与页面名称不匹配 |
| FI-2 | 页面上最显眼的元素是什么？它应该是最重要的吗？ | 最显眼的是装饰性元素而非核心内容 |
| FI-3 | 我知道下一步该做什么吗？有明确的行动入口吗？ | 没有 CTA、所有元素同等突出、不知道从哪里开始 |
| FI-4 | 首屏（不滚动）展示的信息是否足够有价值？ | 首屏全是标题/装饰，有价值的内容需要滚动才能看到 |
| FI-5 | 这个页面和上一个页面的视觉风格是否一致？ | 明显的风格断裂（配色变了、字体变了、布局逻辑变了） |

**记录格式：**
```
首印象 — {页面名}
  FI-1: [PASS|FAIL] {实际观察}
  FI-2: [PASS|FAIL] {最显眼元素是什么，是否合理}
  FI-3: [PASS|FAIL] {CTA 在哪里，是否明确}
  FI-4: [PASS|FAIL] {首屏内容评估}
  FI-5: [PASS|FAIL] {与前页风格对比}
```

## 3. 旅程执行（Step 3 Phase B）

按定义的用户旅程实际操作，每一步记录：

### 3.1 操作前：预期 vs 实际

```
旅程步骤 {N}: {描述}
  预期看到: {用户期望在这一步看到什么}
  实际看到: {截图分析后实际看到什么}
  预期操作: {用户自然会做什么}
  实际操作难度: {操作入口是否容易找到、是否需要滚动/搜索}
```

### 3.2 操作后：反馈评估

每次操作后检查：
- **反馈及时性**：点击后是否有即时反馈？（loading、状态变化、页面跳转）
- **结果符合预期**：操作后的结果是否与用户期望一致？
- **位置感**：操作后用户是否知道自己在哪？（面包屑、高亮导航、标题变化）

### 3.3 旅程完整性评估

旅程走完后回答：
- 任务是否顺利完成？中间有没有卡点？
- 是否需要在页面间来回跳转？
- 有没有"我不知道该点哪里"的瞬间？
- 整体操作步数是否合理？有没有可以省略的步骤？

## 4. 信息架构专项检查

### 4.1 导航体验

| 检查项 | 操作 | FAIL 条件 |
|--------|------|----------|
| 导航可见性 | 查看导航是否在所有页面持续可见 | 导航消失、需要滚回顶部才能切换页面 |
| 当前位置指示 | 检查导航中当前页面是否高亮 | 无高亮、高亮了错误的项 |
| 导航可用性 | 点击每个导航项 | 任何导航项不可点击、点击后无响应 |
| 导航层级 | 评估导航项的分组和排序 | 不相关的功能混在一起、最常用的功能排在最后 |

### 4.2 页面内信息组织

| 检查项 | 评估方式 | FAIL 条件 |
|--------|---------|----------|
| 信息分组 | 相关信息是否放在一起？ | 相关数据散布在页面不同位置 |
| 信息优先级 | 最重要的信息是否在最显眼的位置？ | 关键指标被放在页面底部或折叠中 |
| 信息密度 | 单个区块的信息量是否适中？ | 一个区块内塞了 10+ 个指标，或一个大区块只有 1 个指标 |
| 空间利用 | 页面空间是否被合理使用？ | 大片空白但内容挤在一角；或全部挤满没有呼吸空间 |

### 4.3 跨页面一致性

在所有页面都访问过之后，检查：
- 同类操作（筛选、切换、展开）在不同页面是否用相同的交互模式？
- 同类数据（职业名、分数、日期）在不同页面是否用相同的展示格式？
- 页面之间的数据是否一致？（Dashboard 的汇总数 vs Results 的明细数）
