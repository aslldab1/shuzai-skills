# 验收反馈格式

## 验收通过

```markdown
【VALIDATOR】【验收通过】
Issue #{N}：{title}

## 验收摘要

| 维度 | 结果 | 验证项数 |
|------|------|---------|
| 可访问性 | ✅ 通过 | {n}/{n} |
| 视觉质量 | ✅ 通过 | {n}/{n} |
| 用户旅程 | ✅ 通过 | {n}/{n} |
| 信息架构 | ✅ 通过 | {n}/{n} |
| 功能正确性 | ✅ 通过 | {n}/{n} |
| 设计稿对比 | ✅ 通过 | {n}/{n} |

## 验证详情

### 视觉质量
{全页视觉扫描 + 逐区块检查 + 视口适配结果}
- ✅ {页面}: {布局、间距、对齐评估}

### 用户旅程
{每条旅程的完成情况}
- ✅ 旅程 1 "{名称}": {通过情况}

### 信息架构
{导航、页面编排、操作流程评估}
- ✅ {检查项}: {评估结果}

### 功能验证
{逐条列出每个验证项的 PASS 记录}
- ✅ {验证项}: {简要说明验证了什么}

### 设计稿对比（如适用）
{实现 vs 设计稿的对比结果}
- ✅ {页面}: {一致性评估}

### 截图证据
{关键页面截图的文件路径}
- `step3-A-01-dashboard-fullpage.png` — Dashboard 全页视觉扫描
- `step3-B-01-journey1-start.png` — 用户旅程入口

### WARN 项（如有）
{不影响通过判定的警告项}
- ⚠️ {描述}
```

```markdown
## 问题清单

### ❌ 问题 1：{简短描述}（{CRITICAL/HIGH/MEDIUM}）

**预期行为：** {expected}
**实际行为：** {actual}
**复现步骤：**
1. 打开 {url}
2. 点击 {element}
3. 观察 {what_happened}

**截图/证据：** `{screenshot_path}` 或 {命令输出}
**建议修复方向：** {具体到文件/行号的修复建议，如能推断}

### ❌ 问题 2：{简短描述}（{CRITICAL/HIGH/MEDIUM}）
{同上格式}

## 通过的验证项（{pass_count} 项）
{简要列出已通过的项目，证明不是全盘否定}
- ✅ {验证项}
- ✅ {验证项}

## 修复优先级建议
1. 🔴 {CRITICAL 项} — 阻断性问题，必须首先修复
2. 🟠 {HIGH 项} — 严重影响使用
3. 🟡 {MEDIUM 项} — 影响体验但不阻断
```

## 严重程度定义

| 级别 | 定义 | 示例 |
|------|------|------|
| **CRITICAL** | 功能完全不可用、页面崩溃、数据丢失 | 页面白屏、提交后数据消失、服务启动失败 |
| **HIGH** | 核心功能有明显缺陷、严重影响使用 | 关键按钮无响应、数据显示错误、导航断裂 |
| **MEDIUM** | 影响体验但不阻断核心功能 | 样式错位、文字截断、Console warning、加载无反馈 |

## 写作要求

1. **具体可操作**：每个问题必须描述到 worker 无需追问即可定位和修复的程度
2. **避免模糊**：不说"有些地方不太好"，而说"Dashboard 页面第三个卡片的标题文字被截断，只显示前 8 个字符"
3. **复现步骤完整**：从打开浏览器开始，到问题出现结束，每一步都写清楚
4. **正面也要列出**：在不通过的报告中也列出已通过的项目，让 worker 知道哪些不需要改
5. **修复建议具体**：如果能从操作中推断出问题所在的文件或组件，写出来
