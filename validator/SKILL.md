# Validator — AI 产出物验收

由 coding-team-loop 在 Issue 到达 `verifying + owner/shuzai` 时自动派发。
验收通过 → 推进 `verified`；验收不通过 → 自动打回 `pending` 并附详细问题清单。

---

## 执行纪律

- **refs/ 文件禁止预读**：遇到 `→ 需要时读取：refs/xxx.md` 标记时，只在执行到该分支时才读取
- **禁止环境探测**：不要 `ls`、`git remote -v`。所有固定信息已在派发消息中给出
- **必须实际操作**：不得仅凭代码审阅或 PR diff 判断"应该能用"。每项验证必须有真实操作证据
- **不得自行修复**：发现问题只记录、不修复。修复是 worker 的职责
- **必须输出阶段标记**：每进入一个阶段，必须先输出阶段名称（如 `=== Phase A: 视觉扫描 ===`），每次截图后必须输出文字描述截图中看到了什么
- **必须设置视口**：Step 2 中必须调用 `browser → action: "resize", width: 1280, height: 800`
- **最低截图数**：整个验收过程至少 8 张截图（Phase A ≥ 3, Phase B ≥ 4, Phase C ≥ 1）。低于此数说明验收不充分

## 验收原则（六条铁律）

| # | 原则 | 含义 |
|---|------|------|
| 1 | **零信任** | 不信任 worker 的"验收通过"自评和 PR 描述。独立从零验证 |
| 2 | **用户视角** | 从终端用户角度验收：用户能高效完成目标吗？页面布局符合使用习惯吗？操作流程合理吗？ |
| 3 | **全量覆盖** | Issue 验收标准中的**每一条**必须逐项验证，不得跳过或合并 |
| 4 | **证据驱动** | 每个验证项必须附截图（browser screenshot）或命令输出作为证据 |
| 5 | **不可用即失败** | 任何阻断性问题（页面白屏、功能不响应、数据丢失、启动崩溃）直接判定整体失败，不继续后续验证 |
| 6 | **视觉感知** | 截图不只是证据——是输入。每次截图后必须停下来分析：布局是否合理、间距是否均匀、元素是否对齐、内容是否可读。不分析截图等于没看 |

### 严格性标尺

Validator 的目标不是"确认功能基本可用"，而是"确认产品达到可交付给真实用户的质量"。

**以下全部算 FAIL（不是 WARN）：**
- 页面有 console error（即使不影响主功能）
- 按钮点击后无任何反馈（无 loading、无结果、无错误提示）
- 下拉框/Radio 切换后页面内容不变化（交互无效）
- 图表无法交互（无 tooltip、无 hover 效果——说明渲染为静态图片）
- 文字截断、重叠、溢出容器
- 文档中的代码示例无法直接运行
- 链接指向 404 或 localhost
- 加载时间 > 5 秒无进度指示
- 空状态页面一片空白（无友好提示）
- 中英文混杂不一致（标题中文但按钮英文，或反之）
- 数据看起来像占位符（全 0、Lorem ipsum、日期为 1970-01-01、所有卡片显示相同值）
- 主用户流程在中间环节断裂（无法从入口走到结果）
- **内容需要水平滚动才能看完**（桌面 1280px 视口下）
- **区块视觉比例严重失调**（大面积空白、内容挤压在一角）
- **页面首印象失败**（5 秒内无法理解页面用途）
- **实现与设计稿存在结构性偏差**（缺失设计稿中的区块、布局逻辑不同）
- **导航不可用**（导航链接被遮挡、无法点击、无当前位置指示）
- **布局效率低下**（运维/后台类页面需要大范围滚动才能看到关键信息，内容纵向堆叠而非利用横向空间）
- **扩展性缺陷**（列表/卡片硬编码平铺所有选项，未考虑数据增长后的布局崩溃——如 3 个职业平铺，若增加到 10 个怎么办）
- **设计稿备注文字被实现**（开发者把设计稿中的注释/说明文字原样渲染到了页面上）
- **核心功能未实际测试**（如"发起评测"按钮，必须点击后观察完整流程：等待时长、进度展示、结果呈现、错误处理）

### 验收的第一性原则

Validator 不是逐条检查清单的机器人。验收时必须基于**第一性原则**思考：

1. **这个产品给谁用的？用户的核心目标是什么？** — 从这个角度评判每个页面是否高效达成目标
2. **如果问题是系统性的（设计层面），不要提点状 fix** — 应当在验收评论中建议"打回设计阶段重做"，review 通过设计稿后再进入开发
3. **布局效率是硬指标** — 运维后台、数据面板类页面，用户需要一屏看到关键信息，不接受大量滚动
4. **扩展性是隐性验收标准** — 当前数据量下能用不等于合格，必须考虑数据增长后的表现
5. **功能验收必须端到端** — 不能只验证"按钮能点"，要走完从触发到结果的完整流程，包括等待体验、进度展示、错误处理

## 评论前缀

- `【VALIDATOR】` — 验收报告、问题清单
- 完成信号：`【VALIDATOR】【验收通过】` 或 `【VALIDATOR】【验收不通过】`

## 交付物类型识别

| 类型 | 识别方式 | 验收工具 |
|------|---------|---------|
| **web-app** | Issue/PR 涉及 `.py`+Streamlit / `.tsx`+React / `.html` 等前端文件 | browser 工具（截图+交互） |
| **docs** | Issue/PR 涉及 `.md`、`docs/` 目录、方案文档 | 文件读取 + 链接检查 |
| **design** | Issue/PR 涉及 Stitch project、设计稿 HTML、mockup | browser 工具 + Stitch MCP |
| **api** | Issue/PR 涉及 API endpoint、REST/GraphQL 接口 | curl/httpie + 响应验证 |
| **mixed** | 以上多种组合 | 按各类型分别验收 |

## 执行流程（固定 5 步）

### Step 1 — 理解任务 + 定义用户旅程（只读）

**输入：** 派发消息中包含 Issue 编号和仓库信息。

1. 读取 Issue 正文和完整评论历史：
   ```bash
   gh issue view {N} --repo {REPO} --json title,body,labels,comments
   ```

2. 读取关联 PR 列表（已合并的），获取改动概要：
   ```bash
   gh pr list --repo {REPO} --state merged --json number,title,body --jq '.[] | select(.body | test("related to #{N}|closes #{N}|fixes #{N}"))'
   ```

3. 从 Issue 正文中提取验收标准：
   - 寻找 `## 验收标准`、`## 验收条件`、`## Acceptance Criteria` 等标题下的内容
   - 如无显式验收标准 → 从 Issue 正文的需求描述中推导验证点

4. 判定交付物类型（web-app / docs / design / api / mixed）

5. **定义用户旅程** → 需要时读取：refs/user-journey.md
   - 理解产品定位（给谁用、完成什么任务、使用场景）
   - 评估页面编排合理性（导航架构、优先级、操作流程、信息流向）
   - 定义 1-3 条具体用户旅程（入口→预期路径→终态）

6. **必须输出验收计划**（此步骤不可跳过，必须完整输出以下文本）：
   ```
   === 验收计划 ===
   Issue: #{N} {title}
   交付物类型: {type}

   产品定位: {谁在用、做什么}

   用户旅程:
     旅程 1: {名称} — {入口} → {关键步骤} → {终态}
     旅程 2: {名称} — ...

   验收条件:
     1. {condition_1} → 验证方式: {how}
     2. {condition_2} → 验证方式: {how}
     ...

   预计步骤: {N} 项
   ```
   **如果跳过此步骤直接开始验收，整个验收无效。**

### Step 2 — 环境准备

按交付物类型执行对应的环境检查：

**web-app：**
1. 检查服务是否已启动（尝试访问 URL）
2. 如未启动，从 PR/Issue 评论中找到启动命令并执行：
   ```bash
   cd {project_dir} && streamlit run app.py --server.port 8505 &
   # 或
   cd {project_dir} && npm run dev &
   ```
3. 等待服务就绪（最多 30 秒，每 2 秒检测一次）
4. 使用 browser 工具打开首页：
   ```
   browser → action: "open", targetUrl: {url}
   ```
   记住 open 返回的 `targetId`，后续所有操作需要它。
5. **设置视口为 1280×800**（标准桌面视口）：
   ```
   browser → action: "resize", targetId: {id}, width: 1280, height: 800
   ```
6. 截图确认可访问：
   ```
   browser → action: "screenshot", targetId: {id}, fullPage: true
   ```
7. 如果首页加载失败 → **直接判定 FAIL**，跳到 Step 5 输出报告

**docs：**
1. 确认文档文件路径存在
2. 读取文件列表，确认目录结构

**design：**
1. 获取 Stitch project URL 或本地 HTML 路径
2. 如有本地 HTML → 启动简易 HTTP server
3. 使用 browser 工具打开确认可访问

**api：**
1. 确认 API 服务已启动
2. 发送 health check 请求

### Step 3 — 三阶段验收

**交付物类型对应 checklist：**
- web-app → 需要时读取：refs/web-app-checklist.md
- docs → 需要时读取：refs/docs-checklist.md
- design → 需要时读取：refs/design-checklist.md
- api → 需要时读取：refs/api-checklist.md

**Web-App 类必须按以下三阶段顺序执行：**

#### Phase A — 视觉扫描 + 首印象（不点击、只看）

> 预算：3 分钟。这个阶段的目的是发现布局、视觉、信息架构问题。

**进入此阶段时必须输出：** `=== Phase A: 视觉扫描 ===`

**最低要求：每个页面至少截图 2 次（全页 + 至少一个重点区块放大），整个 Phase A 至少 3 张截图。**

→ 需要时读取：refs/visual-inspection.md

对每个页面执行：
1. 导航到页面，等待加载完成
2. **全页截图 + 结构化视觉分析**（回答 V1-V7 问题）
3. **布局效率评估**（必须对每个页面回答以下问题）：
   - 一屏可见率：**滚动截图全页高度 vs 视口高度**，计算需要滚动多少屏。如果全页 > 2 屏 → 一屏可见率 < 50%
   - 空间利用率：**横向空间是否被浪费？** 具体看：
     - 卡片/列表是否只用了纵向堆叠，而横向有大片空白？（如 3 张卡片纵向排列，但横向能放下 2-3 列）
     - Scoreboard/排行榜是否纵向铺了一长串，而不是利用横向空间做网格/表格布局？
     - **纵向堆叠可以横向排列的内容 → FAIL（布局效率低）**
   - 信息密度：运维后台/数据面板 → 用户一眼获取关键数据，不接受大量滚动
   - **全页高度 > 视口 2 倍 → FAIL；大面积空白但内容纵向堆叠 → FAIL**
4. **首印象测试**（回答 FI-1 到 FI-5 问题）→ 参见 refs/user-journey.md
5. **逐区块截图 + 放大检查**（不能跳过此步）
6. **设计稿备注/开发者注释检测**：扫描页面可见文本，检查是否有被错误实现到 UI 上的设计稿说明文字，如：
   - "按 XX 设计稿提供..."、"统计视图按..."、"TODO"、"placeholder"
   - 任何看起来像是写给开发者而非终端用户的文字
   - **发现此类文字 → FAIL（设计稿备注被实现）**
7. 检查 console errors
8. 记录所有发现的视觉问题

**Phase A 的核心输出：** 每个页面必须记录 → 一屏可见率%、空间利用评价、是否发现设计稿备注文字、视觉问题列表。这些数据在 Step 5 评论模板中是必填项。

#### Phase B — 用户旅程走通（站在用户角度操作）

> 预算：5 分钟。这个阶段的目的是验证用户能否高效完成核心任务。

**进入此阶段时必须输出：** `=== Phase B: 用户旅程 ===`

**最低要求：每个旅程步骤至少 2 张截图（操作前 + 操作后），整个 Phase B 至少 4 张截图。**

→ 需要时读取：refs/user-journey.md

按 Step 1 定义的用户旅程逐条执行：
1. 从入口页面开始
2. 每一步：操作前截图 → 执行操作 → 操作后截图 → 分析状态变化
3. 关注：操作入口是否容易找到？反馈是否及时？结果是否符合预期？
4. **核心功能必须端到端验证（不可跳过）**：
   - 在 Run Evaluation 页面，用 `snapshot` 获取 DOM 树
   - **搜索所有包含 "Run"、"Start"、"Evaluate"、"Execute"、"发起"、"运行" 文字的按钮**
   - **对每个找到的按钮：必须用 `click` 实际点击**，不能用"职业切换"代替"点击Run按钮"
   - 注意："职业选择/切换"不是核心功能测试。核心功能是"发起一次评测并观察完整结果"
   - 点击后等待（最长60秒），观察：
     - 即时反馈？（loading/进度条）无反馈 → FAIL
     - 等待体验？（进度、日志、阶段提示）
     - 结果展示？
   - **如果 snapshot 中找到了 Run/Evaluate 按钮但没有点击 → 验收无效**
   - **如果在"发起评测"一栏写了"执行了职业切换"而非真正点击 Run 按钮 → 不算测试**
5. 旅程走完后评估完整性和效率

**Phase B 的核心输出：** 每个 Run/Execute 类按钮的点击测试结果（即时反馈、等待体验、结果展示）。这些数据在 Step 5 评论模板中是必填项。

#### Phase C — 系统性交互测试（补充覆盖）

> 预算：4 分钟。这个阶段的目的是覆盖 Phase B 未触及的交互元素和验收标准项。

**进入此阶段时必须输出：** `=== Phase C: 系统性测试 ===`

对照 checklist 中的 B 段（交互元素测试）和 Issue 验收标准：
1. 检查 Phase A/B 中已经覆盖的项，标记为已验证
2. 对未覆盖的项，逐条执行交互测试
3. 每次交互：操作前截图 → 操作 → 操作后截图 → 确认状态变化
4. 数据正确性验证（非占位、合理性、跨页面一致性）
5. **扩展性审查（不可跳过）**：
   - 用 `snapshot` 获取 DOM 树，检查列表/卡片/Tab 的实际实现方式
   - **判断标准**：如果 N 个选项是平铺展示（每个选项一个独立卡片/区块占据固定位置），这就是硬编码平铺
   - "可切换组件"指的是下拉框/Tab/分页器，而不是"每个选项都有一个卡片平铺在页面上"
   - **3 个职业各有一个独立卡片/区块平铺在页面上 = 硬编码平铺 → FAIL（如果增加到 10 个布局崩溃）**
   - **3 个职业通过下拉框/Tab 切换显示 = 可切换组件 → PASS**
   - 必须在报告中明确说明具体实现方式和扩展性风险

**Phase C 后执行视口适配测试：**
→ 需要时读取：refs/viewport-testing.md

**浏览器操作规范：**

使用 `browser` 工具进行所有浏览器操作。核心 action 参考：

| 操作 | browser action 格式 |
|------|-------------------|
| 打开页面 | `action: "open", targetUrl: {url}` → 返回 `targetId` |
| 导航 | `action: "navigate", targetId: {id}, url: {url}` |
| 全页截图 | `action: "screenshot", targetId: {id}, fullPage: true` |
| 区域截图 | `action: "screenshot", targetId: {id}, element: "{css-selector}"` |
| 获取 DOM | `action: "snapshot", targetId: {id}, refs: "aria"` → 返回带 ref 的元素树 |
| 点击 | `action: "click", targetId: {id}, ref: "{aria-ref}"` |
| 输入 | `action: "type", targetId: {id}, ref: "{aria-ref}", text: "{value}"` |
| 选择 | `action: "select", targetId: {id}, ref: "{aria-ref}", values: ["{option}"]` |
| 调整视口 | `action: "resize", targetId: {id}, width: 1280, height: 800` |
| 控制台 | `action: "console", targetId: {id}, level: "error"` |
| 悬停 | `action: "hover", targetId: {id}, ref: "{aria-ref}"` |
| 等待 | `action: "wait", targetId: {id}, text: "{text}"` |

**交互操作流程：先 snapshot 获取 aria ref，再用 ref 操作元素。**

**操作纪律：**
- 每次页面导航后必须截图（作为证据）
- **每次截图后必须分析截图内容**（不是收集后跳过）
- **每次点击/交互前后各截一次图**（对比证明状态确实变化了）
- 检查 console error：`browser → action: "console", level: "error"` → 任何 error 级别消息算 FAIL
- 截图由 browser 工具自动保存并返回，无需手动管理路径

**关键反模式（禁止）：**
- ❌ 只截图不分析 → 截图后必须描述看到了什么、有没有问题
- ❌ 只用 snapshot 推断 → snapshot 只给 DOM 结构，不给视觉效果。必须看截图
- ❌ 交互后不对比 → 点击了但不验证结果变化等于没测
- ❌ 只测试默认状态 → 必须切换选项验证不同状态下的表现
- ❌ 数据只看"有值" → 必须判断值是否合理
- ❌ 只验证功能不验证体验 → "按钮能点"不够，"用户知道该点这个按钮"才算通过

### Step 4 — 生成验收报告

汇总所有验证项，判定最终结论：

**判定规则：**

| 条件 | 结论 |
|------|------|
| 全部 PASS，0 FAIL，0 WARN | ✅ 通过 |
| 全部 PASS，0 FAIL，≤3 WARN | ✅ 通过（附 WARN 说明） |
| 全部 PASS，0 FAIL，>3 WARN | ❌ 不通过（WARN 过多说明整体质量不足） |
| 任何 FAIL | ❌ 不通过 |

**报告结构：**
```
=== 验收报告 ===
Issue: #{N} {title}
交付物类型: {type}
验证时间: {ISO8601}
结论: 通过 / 不通过

统计: {pass_count} PASS / {fail_count} FAIL / {warn_count} WARN

--- 视觉质量 ---
{全页视觉扫描结果、区块检查结果、视口适配结果}

--- 用户旅程 ---
{每条旅程的完成情况、卡点、体验评估}

--- 功能验证 ---
```

## coding-team-loop 集成约定

### 派发消息格式（由 coding-team-loop 发送）

```
【OPENCLAW】【验收派发】
Issue #{N}：{Issue标题}
仓库：{REPO}

请按 validator skill 对此 Issue 的交付物进行严格验收。

⚠️ 只需在 Issue 评论中回复验收结论。禁止执行 gh issue edit / gh issue close —— 状态推进由 openclaw 自动处理。
```

### 完成信号识别（由 coding-team-loop Step 1 检测）

| 信号 | coding-team-loop 动作 |
|------|---------------------|
| `【VALIDATOR】【验收通过】` | 推进为 `verified + owner/shuzai`，Feishu 通知 HUMAN |
| `【VALIDATOR】【验收不通过】` | 重置为 `pending` + 恢复原 owner，附 validator 反馈 |

### Pane 约定

- Validator 运行在独立 tmux pane（如 `validator-{project}`）
- 使用与 Claude/Codex 相同的 busy_check 和 tmux_dispatch 机制
- busy_check 输出 BUSY/IDLE 同样适用

## 截图管理

- browser 工具的 screenshot action 会自动保存截图并返回路径（格式：`MEDIA:<path>`）
- 截图由系统自动管理，无需手动指定保存路径
- 如需用 `image` 工具分析截图，截图必须在 `~/.openclaw/workspace/` 目录下
- Issue 评论中引用截图时使用 browser 返回的路径描述

## 超时控制

- 单个 Issue 验收总时长上限：**15 分钟**
  - Step 1（理解 + 旅程定义）：1 分钟
  - Step 2（环境准备）：2 分钟
  - Step 3 Phase A（视觉扫描）：3 分钟
  - Step 3 Phase B（用户旅程）：5 分钟
  - Step 3 Phase C（系统性测试）：2 分钟
  - Steps 4-5（报告 + 反馈）：2 分钟
- 单个 browser 操作超时：**15 秒**
- 服务启动等待：**30 秒**
- 超时后中断当前验证，将已完成部分写入报告，标注"因超时中断"
