# 进度报告格式（飞书/钉钉）

每轮执行完成后，必须按本模板组织输出内容。这是返回给 cron 系统的 summary，会直接推送到飞书，HUMAN 只看这段内容。

## 严格要求

- 用正常汇报工作的口语，想象你在给非技术同事发消息
- 禁止出现：Issue 编号单独使用（如 `#51`）、label 名称（pending/in-progress）、step 编号、技术术语（P1、dispatch、pane、hash、owner/shuzai、run_lock 等）
- 每条必须写完整：「需求名称」＋当前状态一句话，缺一不可
- 如有 PR 链接等具体信息，用"查看详情"之类自然语言引导

## 模板（逐字段对照填写）

```
【开发进度 YYYY-MM-DD HH:MM】

正在处理：
• 「{Issue标题}」— {当前在做什么，例如：Claude 正在阅读需求和构思方案}
• 「{Issue标题}」— {例如：Codex 正在实现登录功能}

需要你操作：
• 「{Issue标题}」— {需要做什么，例如：方案已完成等你确认，看一下是否符合预期}
• 「{Issue标题}」— {例如：功能已合并，请验收一下效果}

排队中：
• 「{Issue标题}」— 还没开始，下轮自动启动

本轮变化：{一句话描述变化，例如：把前端设计任务派给了 Claude 开始执行；无变化则写「无变化」}

——————————————————————————————

执行详情（调试用）：

信号读取：
• 扫描到 {N} 个需求，{M} 个待审代码
• 「{Issue标题}」状态：{用口语描述当前状态，例如：已完成验收，球在你手里}
• 「{Issue标题}」状态：{例如：排队中，由 Claude 负责}

Worker 状态：
• Claude：{空闲，可以接任务 / 忙碌，正在处理「XX」}
• Codex：{空闲，当前没有实现类任务需要处理 / 忙碌，正在实现「XX」}

派发过程：
• Claude 侧：{描述选中了哪个任务、为什么选它、派发结果}
• Codex 侧：{描述选中了哪个任务，或无待处理任务本轮跳过}

状态更新：
• {口语描述本轮写入了什么状态变更，例如：需求状态已从排队更新为执行中，写入了派发记录}
• {例如：进度记录已刷新}
```

## 分段说明

| 分段 | 对应状态 | 无内容时 |
|------|---------|---------|
| 正在处理 | 执行中的任务 | 省略整段 |
| 需要你操作 | 等 HUMAN 确认的任务 | 省略整段 |
| 排队中 | 待派发的任务 | 省略整段 |

- 间隔符上方是给 HUMAN 快速看结论的摘要，下方是完整执行链路供 review
- 执行详情部分同样禁止出现 label 名称、Step 编号等技术术语，全部用口语描述
- 所有已验收且已通知过的需求不重复出现

## 错误示例（以下写法严格禁止）

```
❌ Open Issue：2 个（#51、#54），都在排队且由 Claude 负责
❌ Claude 侧 P1~P4、Codex 侧 P1~P2 均跳过
❌ owner/shuzai 队列为空
❌ 未发生 Issue 状态变化，因此不发送进度播报
❌ Worker 状态检查：Claude pane：BUSY
❌ 本轮 coding-team-loop 巡检已执行，结果如下：
```

## 正确写法

```
✅ 「Harness Engineering 规范落地」— 排队中，由 Claude 负责，下轮自动启动
✅ Claude 和 Codex 都在忙，本轮没有派发新任务
✅ Claude：忙碌，正在处理上一个任务
✅ 本轮无变化
✅ 【开发进度 2026-03-29 13:12】
```
