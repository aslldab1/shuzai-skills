openclaw 协调 Claude + Codex 双 Worker 开发循环 — 设计文档
v2 · 2026-03-29
一

设计原则

任务是核心
多任务并行于不同阶段，而非单一状态机串行推进。
双 Worker 独立调度
Claude 和 Codex 互不阻塞，同一轮可同时向两侧派发。
GitHub Issues 是唯一任务数据库
HUMAN 创建 Issue，openclaw 维护 label，AI 不自行发明任务。
交付有明确定义
Claude 验证主干使用流程通过才算已交付，PR 合并不等于交付。
评论前缀区分身份
同一 git 账号，通过 【OPENCLAW】/【CLAUDE】 前缀识别来源。Claude 所有 Issue 评论必须以 【CLAUDE】 开头。
原文转发，禁止改写
Fix 派发时 review 意见必须完整逐字转发，防止信息失真。
完整信息驱动判断
每次派发前读取 Issue 完整评论历史 + worker pane 内容，交给 LLM 综合推断当前状态，不依赖单一脆弱信号。
派发评论是状态锚点
【OPENCLAW】派发评论是最可靠的历史记录，必须是派发动作的第一步，确保即使后续中断也能追溯。

五

每轮执行结构（固定 3 步）

1

读取信号（只读）

Issues
读取所有 open Issues 及 label；检测 stale in-progress（worker 空闲 + 无关联 PR）
对于每个 pending + owner/claude/codex 的 Issue，额外读取最近 10 条评论（含时间戳），供 Step 2/3 判断使用
PRs
扫描 PR body 中 closes/fixes #N；自动推进 in-progress → needs-review；检测 commit > review 时间
Pane
抓取 Claude / Codex pane 最后 30 行内容：判断忙碌状态 + 感知 worker 最近的执行结果 tmux capture-pane -p -t {pane} -S -30
Lock
读取 run_lock，若为 true 则跳过本轮（防并发）
2

处理 Claude 侧（忙碌则跳过）

P1
有 needs-review Issue → 派发 review 请求，通过后直接 merge refs/review-and-merge.md
P2
有 verifying Issue → 派发验收请求 refs/deliverable-verify.md
P3
有 pending + owner/claude → 取编号最小，综合判断后派发
读取该 Issue 完整评论历史 + Claude pane 内容，LLM 综合推断：任务是否已被处理、是否有未响应的 HUMAN 反馈、应使用标准格式还是带反馈格式。refs/task-dispatch.md
P4
有 pending + 无 owner label → 派发路由请求，Claude 判断 owner 或拆解 epic
P5
以上均无 → 本轮无动作
3

处理 Codex 侧（忙碌则跳过）

P1
有 changes-requested Issue → 派发 fix 请求，原文转发 review 意见 refs/fix-dispatch.md
P2
有 pending + owner/codex → 取编号最小，综合判断后派发 读取该 Issue 完整评论历史 + Codex pane 内容，LLM 综合推断。refs/task-dispatch.md
P3
以上均无 → 本轮无动作
全局异常： 两侧均无动作时 — 有 in-progress 但 worker 空闲超 2h → Feishu 通知；所有 Issue verified + 无 pending → 通知"请下发新任务"（去重）；连续失败 3 次 → Feishu 通知
六

派发动作顺序（必须严格按序）

每次决定派发后，以下三步必须按顺序执行。顺序不可颠倒，原因见各步说明。

1
写 【OPENCLAW】 派发评论到 Issue
gh issue comment {N} --body "【OPENCLAW】已将此任务派发给 @{worker}…"
⚑ 必须最先做：这是唯一持久化的派发记录。即使后续步骤失败，下一轮也能通过此评论感知「曾经派发过」，避免状态完全丢失。
2
更新 Issue label：pending → in-progress
gh issue edit {N} --add-label in-progress --remove-label pending
⚑ 评论已写，此时改 label 是安全的。即使第 3 步失败，下一轮看到 in-progress + 派发评论 + worker 空闲 → 走 stale 自愈逻辑重新派发。
3
tmux 发消息给 worker
bash scripts/tmux_dispatch.sh {pane} "{payload}"
⚑ 最后执行：若 dispatch=failed，不影响已写入的评论和 label，下一轮 stale 检测自动恢复。
中断容错： 任何步骤中断后，下一轮 LLM 通过读取 Issue 评论历史 + pane 内容，可以准确判断执行到了哪一步，采取正确的恢复动作。
七

状态感知与 LLM 判断（P3 核心逻辑）

遇到 pending + owner/claude 或 owner/codex 的 Issue 时，不使用规则枚举，而是将完整信息交给 LLM 综合判断。

输入信息
信息源读取内容作用
Issue 评论历史 最近 10 条，含时间戳和发送者前缀 判断最后一次派发时间、是否有 HUMAN 反馈、worker 是否已回复
Worker pane 最近 30 行 感知 worker 当前是否在执行、最近是否完成了相关任务
判断维度（LLM 综合推断，不是机械规则）
任务是否已处理？ — pane 显示 worker 最近说"已完成/已处理"，且评论中有对应的 【CLAUDE】 回复 → 工作已做，只需修正 label，不重新派发
是否有 HUMAN 反馈？ — 找最近一条 【OPENCLAW】派发评论，其后是否存在无系统前缀的评论 → 有则附带反馈内容派发；无则标准派发
payload 编码是否正确？ — 从 gh issue view --json 读取评论时，必须通过 jq -r 解码，确保中文为原始 UTF-8，不含 \uXXXX 转义序列
Claude Issue 评论规范
Claude 发布到 Issue 的所有评论必须以 【CLAUDE】 开头。这是系统区分 HUMAN 评论与 Claude 回复的唯一依据，缺少前缀会导致下一轮将 Claude 的回复误判为新的 HUMAN 反馈。

九

Worker 忙碌判断

抓取 tmux pane 最近 30 行内容判断，不使用时间阈值。Claude 和 Codex 通用相同规则。
忙碌 — 有进行时内容
  • …ing 类文本：Thinking… / Running… / Writing… / Reading…
  • spinner 字符：⠋ ⠙ ⠹ ⠸ 等
  • 工具调用行：⏺ Bash( / ⏺ Read( / ⏺ Write( 等
空闲 — 等待输入
  • 最后一个非空行是输入提示符
  • ❯ — Claude Code 默认提示符
  • $ 或其他 shell 提示符（Codex 环境）
tmux capture-pane -p -t {session:window.pane} -S -30
若 pane 不存在 → 视为会话缺失，参考 refs/session-recovery.md
十

Memory 数据结构

workers: claude: last_dispatched_at: "" # ISO8601 last_issue_number: null codex: last_dispatched_at: "" last_issue_number: null notifications: all_verified_notified_at: null # 防止重复通知 last_feishu_report_hash: "" # Issue label 状态 hash，内容不变时跳过报告 run_lock: false # 防并发：本轮执行中设为 true，完成后清除
Memory 只存调度状态，任务详情（Issue 内容、PR 链接、review 意见）全部存在 GitHub Issues，不在 Memory 中冗余。

十二

新旧方案对比

旧方案（规则枚举 + 脆弱信号）

  • 用"最新评论前缀"单一信号判断状态
  • 派发后动作分散，中断即状态不一致
  • 评论内缺少前缀 → 被误判为 HUMAN 反馈
  • pane 只读 10 行，感知不到 worker 执行结果
  • 派发评论写在最后，中断后无历史锚点
  • 中断后无自愈，重复派发形成死循环
  • JSON 评论文本未解码，\uXXXX 乱码发送

新方案（完整信息 + LLM 综合判断）

  • 读取完整评论历史 + pane 30 行，LLM 综合推断
  • 派发评论永远最先写，是不变的历史锚点
  • Claude 评论强制 【CLAUDE】 前缀，身份识别可靠
  • pane 内容作为辅助信号感知 worker 真实状态
  • 中断后下轮可准确判断执行到哪一步并恢复
  • 评论文本通过 jq -r 解码，确保 UTF-8 正确传递
  • LLM 判断替代规则枚举，处理边缘 case 更健壮
维度 旧方案 新方案
状态判断依据单一信号（最新评论前缀）Issue 评论历史 + pane 内容（LLM 综合）
派发记录写在动作最后，易丢失写在最先，是不变锚点
中断恢复状态不一致，重复派发下轮 LLM 根据历史推断，精准恢复
评论身份识别依赖前缀，缺失即误判强制前缀 + pane 双重验证
中文编码直接嵌 JSON，\uXXXX 乱码jq -r 解码，原始 UTF-8
任务模型单队列串行多任务并行多阶段
worker 调度互斥等待独立调度，同轮双发
交付定义PR mergeClaude 验收主干流程通过
